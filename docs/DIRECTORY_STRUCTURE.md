# HeyFoS Directory Structure

## Overview
HeyFoS uses a hierarchical directory structure to organize user data, processing sessions, and results.

## Directory Layout

```
users/
├── <userID>/                    # Unique user identifier (UUID)
│   ├── <sessionID-1>/          # Processing session/job (UUID)
│   │   ├── upload/             # Input images for this job
│   │   │   ├── image1.TIF
│   │   │   ├── image2.TIF
│   │   │   └── ...
│   │   └── result/             # Output/processed images
│   │       └── stacked.tif
│   │
│   ├── <sessionID-2>/          # Another processing job
│   │   ├── upload/
│   │   └── result/
│   │
│   └── <sessionID-3>/
│       ├── upload/
│       └── result/
```

## User Identification

- **userID**: Generated once per browser and stored in `localStorage`
- **Persistence**: Survives browser refresh and tab closing
- **Format**: UUID v4 (e.g., `fa45dc70-0f89-4b95-8758-615792e910be`)

## Session/Job Identification

- **sessionID**: Generated **each time** user clicks "Start Processing"
- **Purpose**: Groups one set of input images with their processing result
- **Lifecycle**: One sessionID = One processing job/batch
- **Format**: UUID v4 (e.g., `099b07ee-1b61-41df-9691-481fa3d1fe38`)

## Upload Directory (`upload/`)

**Purpose**: Stores original input images uploaded by user

**Contents**:
- RAW image files (CR2, NEF, ARW, CR3, DNG)
- TIFF files (TIF, TIFF)
- One upload directory per processing job

**Behavior**:
- Created when user uploads files via "Start Processing"
- Files are saved with original filenames
- Directory persists after processing completes

## Result Directory (`result/`)

**Purpose**: Stores processed/stacked output images

**Contents**:
- Final focus-stacked image (typically TIFF format)
- Depth maps
- Any intermediate processing results

**Behavior**:
- Created when processing job starts
- Populated when processing completes
- Available for download via frontend UI

## Workflow Example

1. User opens HeyFoS in browser
   - `userID` generated/loaded from localStorage
   
2. User selects 13 TIF files and clicks "Start Processing"
   - New `sessionID` generated: `abc123...`
   - Directory created: `users/<userID>/abc123/upload/`
   - Files uploaded to upload directory
   
3. Processing starts
   - Directory created: `users/<userID>/abc123/result/`
   - Focus stacking algorithm runs
   - Output saved to result directory
   
4. User starts another job with different images
   - New `sessionID` generated: `def456...`
   - New directories: `users/<userID>/def456/upload/` and `/result/`
   - Previous session (`abc123`) remains intact

## API Behavior

### Upload Endpoint: `POST /api/stacks/create`
- **Headers**: 
  - `X-User-ID`: User's UUID
  - `X-Session-ID`: New session UUID (generated by frontend)
- **Action**: Creates `users/<userID>/<sessionID>/upload/` and saves files

### Process Endpoint: `POST /api/stacks/{stackID}/process`
- **Action**: Creates `users/<userID>/<sessionID>/result/` and runs processing

### Status Endpoint: `GET /api/jobs/{jobID}/status`
- **Returns**: Processing status (uploading, processing, completed, failed)

### Result Endpoint: `GET /api/jobs/{jobID}/result`
- **Returns**: Processed image from `result/` directory

## Benefits of This Structure

1. **Isolation**: Each processing job is completely isolated
2. **History**: User can see all previous processing sessions
3. **Reproducibility**: Original inputs preserved alongside results
4. **Cleanup**: Easy to delete old sessions without affecting others
5. **Multi-device**: Same userID works across devices (if localStorage synced)

## Cleanup Strategy (Future)

- Delete sessions older than 30 days
- Allow user to manually delete individual sessions
- Keep only N most recent sessions per user
- Compress old sessions to save disk space
